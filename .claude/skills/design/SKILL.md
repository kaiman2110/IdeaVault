仕様の設計・実装計画を行う。引数 = $ARGUMENTS

## フェーズ 1: コンテキスト収集

1. MEMORY.md 配下のメモファイルを読み、既存の仕様・アーキテクチャを把握する
2. $ARGUMENTS が Issue 番号の場合:
   - `gh issue view $ARGUMENTS --repo kaiman2110/IdeaVault` で Issue の内容を確認する
3. $ARGUMENTS がテーマの場合:
   - 関連する既存コードやリソースを探索する
4. 関連する既存の実装・パターンをコードベースから調査する
5. `docs/ideas.md` に関連するアイデアがあれば参照する

## フェーズ 1.5: リサーチ連携（任意）

- `docs/research/` に関連するリサーチレポートがあれば読み込み、設計の参考にする
- リサーチが不足していると感じたら「`/research-game {テーマ}` で先に調査しますか？」と提案する

## フェーズ 2: 仕様設計

ユーザーと対話しながら以下を設計する:

1. **概要**: 何を実現するか（1-2文）
2. **影響範囲**: ユーザー体験がどう変わるか
3. **システム仕様**: 具体的なパラメータ・挙動・UI
4. **既存システムとの連携**: シグナル・グローバル状態・データとの関係
5. **参考事例からの知見**: `docs/research/` のレポートがあれば活用
6. **リスク・トレードオフ**: 実装コスト、バランス懸念、技術的制約

## フェーズ 3: 実装計画

仕様が固まったら実装計画を作成する:

1. **Issue 分割**: 大きい機能は子 Issue に分割する
   - 各 Issue は 1 ブランチで完結する粒度にする
   - 依存関係と実装順序を明記する
2. **各 Issue の概要**:
   - 変更対象ファイル
   - 新規リソース / シグナル / グローバル状態の有無
   - テスト方針
3. **優先度ラベルの提案**: `priority/high` / `medium` / `low`

## フェーズ 4: Issue 作成（ユーザー承認後）

ユーザーが承認したら:

1. `gh issue create --repo kaiman2110/IdeaVault` で Issue を作成する
2. MEMORY.md の Issue テーブルを更新する

## 注意事項

- 各フェーズでユーザーに確認を取りながら進める（一方的に決めない）
- 既存のアーキテクチャ原則（CLAUDE.md）に従う
- Issue 作成はユーザーの明示的な承認後にのみ行う
- **設計承認済みの Issue は `/start-issue` で再度の計画確認をスキップする**
